# Architecture

This is a technical description of how Nullify is put together: the contracts, the circuit, and
how the pieces talk to each other. Some components described here are implemented and tested
locally; others are still skeletons. Each section says which.

## Components

| Part | Where it runs | State |
|------|---------------|-------|
| Shielded pool | on-chain | implemented, tested locally |
| Poseidon Merkle tree | on-chain | implemented, tested locally |
| PLONK verifier | on-chain | generated from circuit, tested locally |
| Withdrawal circuit | client | implemented, tested locally |
| TypeScript SDK | client | implemented |
| Relayer node | off-chain | implemented |
| Cross-chain attestor | on/off-chain | skeleton only |

## The deposit side

A deposit is just a commitment written into a Merkle tree.

The client picks two random field elements, `nullifier` and `secret`, and computes:

```
commitment = Poseidon(nullifier, secret)
```

That commitment is the only thing that goes on-chain. The amount is fixed (the pool is a single
denomination), so deposits are indistinguishable by value. The tree is an incremental Merkle tree
of depth 32, using the same Poseidon hash on-chain and in the circuit, which is what lets a proof
built off-chain verify against the on-chain root. The pool keeps the last 30 roots so a proof built
against a slightly stale root still checks out.

## The withdrawal side

A withdrawal proves three things in zero knowledge, and reveals only what it has to.

Public inputs: `root`, `nullifierHash`, `recipient`, `relayer`, `fee`.
Private inputs: `nullifier`, `secret`, and the Merkle path (`pathElements`, `pathIndices`).

The circuit checks:

1. `commitment = Poseidon(nullifier, secret)` is a leaf reachable from `root`.
2. `nullifierHash = Poseidon(nullifier)`.
3. `recipient`, `relayer`, and `fee` are bound into the proof so a relayer cannot rewrite them.

One detail that matters for soundness: every path-selector bit is constrained to be 0 or 1
(`b * (1 - b) == 0`). Skip that and a prover can forge a Merkle path and drain the pool. It is in
the circuit and covered by the negative tests.

## Contracts

```
NullifyPool            deposit(commitment), tree + roots, holds funds
NullifyVerifier        checks PLONK proof + root, then tells the pool to pay
PlonkVerifier          generated by snarkjs from the circuit
CrossChainAttestor     mirrors a root from one chain to another (skeleton)
```

The verifier is the only address allowed to call the pool's payout function. It refuses a withdraw
if the nullifier is already spent, if the root is unknown, or if the proof does not verify. Only
after all three pass does the pool release `denomination - fee` to the recipient and `fee` to the
relayer, in the same transaction.

## Cross-chain, and what is honest about it today

The intended design: each chain runs its own pool, and an attestor mirrors a committed root from
the source chain to the destination so a withdrawal on chain B can reference deposits made on chain
A. That widens the anonymity set across chains instead of fragmenting it per chain.

What exists today: the attestor is an M-of-N signer skeleton. It is not wired between live chains,
and the single-chain path is the only one exercised by the tests. The honest framing is EVM-first,
single-chain working locally, cross-chain still ahead.

## Trust assumptions

Worth being upfront about these.

- The PLONK trusted setup must be honest. The current one is a development setup and is not safe
  for real funds. A real ceremony comes before mainnet.
- For cross-chain, an honest attestor majority is assumed. A dishonest majority could forge a root
  (a fund-safety problem), but it still cannot deanonymize users, since it only ever sees roots.
- At least one honest relayer needs to be reachable. If every relayer refuses, the proof is still
  reusable until the nullifier is spent, so the user can retry elsewhere.
- The contracts are unaudited.
